# FMC Paperless v1.0.0 — Architecture & Implementation Plan

## 1. Tech Stack Decision

| Layer | Choice | Rationale |
|---|---|---|
| Framework | React 18 + TypeScript 5 | Requirement |
| Build | Vite 6 | Fast HMR, modern ESM, good SPA support |
| State | Zustand | Lightweight, tree mutation friendly, middleware for persist/undo |
| Styling | Tailwind CSS 4 | Rapid styling for light-blue rounded theme, responsive utilities |
| Scramble | `cstimer_module` | Requirement specified. `getScramble('333')` + `getImage(scr, '333')` → SVG. Wrapped in Web Worker for browser use. |
| Move Processing | `cubing/alg` + custom | `cubing/alg` handles parse/invert. Custom simplifier for FMC-specific cancellation rules (opposite face pairs, wide separation). |
| Tree View | Custom component | FMC tree is domain-specific (mind-map branching, complex node rendering). General tree libs add overhead without matching the interaction model. |
| ID Generation | `nanoid` | Lightweight unique IDs for tree nodes |
| Timer | Custom hook | Simple interval-based, no library needed |

## 3. Data Model Design

```typescript
// ============ core/moves/types.ts ============

type Face = 'R' | 'L' | 'U' | 'D' | 'F' | 'B' | 'x' | 'y' | 'z';
type Modifier = 'none' | 'prime' | 'double';

interface Move {
  face: Face;
  modifier: Modifier;
  wide: boolean;    // true for Rw, Lw, etc. Always false for x/y/z
}

interface MoveSequence {
  moves: Move[];
}

// ============ core/solution/types.ts ============

type Priority = 'high' | 'medium' | 'low' | 'none';
type NodeColor = string; // Tailwind color class or hex

interface Insertion {
  id: string;
  placeholder: string;       // "#" or custom symbol
  moves: string;             // Raw move string "U2 B2 U2 L2 B2 U2 B2 L2"
  type: 'normal' | 'wide';   // normal = direct insert, wide = w/w2/' replacement
}

interface SolutionNode {
  id: string;
  moves: string;             // Raw move string "D L B' U B' L'"
  label: string;             // "EO", "4A1", "HTR", "FINISH", etc.
  annotation: string;        // User notes (unlimited length)
  priority: Priority;
  color: NodeColor;          // Custom background
  children: SolutionNode[];  // Branch solutions (ordered)
  insertions: Insertion[];   // Sub-solution modules attached to this node
  isExpanded: boolean;
  moveCount: number;         // Computed: number of moves (for display like "4/6")
  cumulativeCount: number;    // Computed: cumulative moves up to this node
}

// ============ core/scramble/types.ts ============

interface ScrambleData {
  text: string;              // "R U2 R' F2 L' B D' ..."
  image: string;             // SVG string from cstimer_module
  seed?: string;             // Optional seed for reproducibility
}

// ============ Store types ============

interface SolveSession {
  id: string;
  scramble: ScrambleData;
  solutionTree: SolutionNode;
  arrangementBoard: string;  // Manual board text content
  timer: TimerState;
  createdAt: number;         // Unix timestamp ms
  updatedAt: number;         // Unix timestamp ms
}

interface TimerState {
  mode: 'countdown' | 'countup';
  initialSeconds: number;    // Default: 3600 (60 min)
  startedAt: number | null;  // Unix timestamp when started
  elapsedAtPause: number;    // Elapsed seconds when paused
  isRunning: boolean;
  isOvertime: boolean;
}

interface KeyboardLayoutConfig {
  position: 'bottom' | 'left' | 'right' | 'float';
  isVisible: boolean;
  rows: KeyboardRow[];
}

interface KeyboardRow {
  id: string;
  keys: string[];            // Key identifiers: "R", "R'", "R2", "Rw", etc.
}

interface AppSettings {
  theme: 'light-blue';
  keyboard: KeyboardLayoutConfig;
  timer: {
    defaultMinutes: number;  // Default: 60
    defaultMode: 'countdown' | 'countup';
  };
  autoSave: boolean;
}
```

## 4. Core Algorithms

### 4.1 Move Simplification (simplifier.ts)

The FMC-specific cancellation rules:

1. **Same-face cancellation**: Consecutive moves on the same face merge.
   - `R R` → `R2`, `R R2` → `R'`, `R R'` → (cancelled), etc.
   - After simplification, each face has 0 or 1 move.

2. **Opposite face pair cancellation**: When two opposite-face moves are adjacent (with possible non-opposite moves between), they can sometimes cancel:
   - `U D` → can cancel to identity (both same modifier)
   - `U' D2` + `U'` → `D2` (U' and U' cancel, D2 remains)
   - But `U D U` → `U2 D` or `D U2` depending on convention

3. **Wide separation**: `Uw` and `D` do NOT cancel. Only `Uw` + `Dw` or `U` + `D`.

4. **Axis rotations (x, y, z)**: Treated as whole-cube rotations, NOT face moves. They don't cancel with face moves.

Algorithm:
```
1. Parse all moves into Move objects
2. First pass: cancel consecutive same-face moves (reduce to 0-1 per face)
3. Second pass: check opposite face pairs (U-D, F-B, L-R, and their wide variants)
   - Only cancel if both are same modifier or can combine
4. Repeat until no more cancellations possible
```

### 4.2 Solution Compilation (compiler.ts)

```
Input: SolutionNode tree
Output: CompiledMoveSequence

Algorithm:
1. Traverse tree (pick the selected/active path, or main path)
2. For each node in order:
   a. Parse the node's move string
   b. Resolve insertions: replace placeholders (#) with insertion moves
      - For 'wide' insertions: replace each move with its wide variant
   c. Add to accumulated sequence
3. Apply FMC bracket convention:
   - Non-parenthesized segments: keep in order
   - Parenthesized segments: reverse their internal order
4. Concatenate all segments
5. Run move simplification on the full sequence
6. Return compiled result with move count
```

### 4.3 Shadow Solution (shadow.ts)

```
Input: SolutionNode
Output: SolutionNode (sibling)

Algorithm:
1. Clone the source node
2. For each move in the sequence:
   - If modifier is 'prime' → change to 'none' (e.g., R' → R)
   - If modifier is 'none' → change to 'prime' (e.g., R → R')
   - If modifier is 'double' → keep as 'double' (R2 stays R2)
3. Label as "Shadow" or inherit label
4. Return as sibling node
```

### 4.4 Continuous Click Cycling (keyboard/cycling.ts)

```
State per key: { lastMove: string, clickCount: number }

On click(key):
  if key matches lastMove:
    clickCount++
    if clickCount === 1: append key (e.g., R → show R)
    if clickCount === 2: replace with 2x (R → R2)
    if clickCount === 3: replace with prime (R → R')
    if clickCount === 4: remove (R → empty)
  else:
    lastMove = key, clickCount = 1
    append key

Timeout (500ms): reset clickCount
```

## 5. State Management Design (Zustand)

```typescript
interface SolveStore {

  sessions: SolveSession[];

  // Scramble actions

  regenerateScramble: () => void;

  loadFromLocalStorage: () => void;
```

### Store: treeStore
```typescript
interface TreeStore {
  // Actions (all operate on active session's tree)
  addNode: (parentId: string | null, afterId: string | null, node: Omit<SolutionNode, 'id' | 'children' | 'isExpanded' | 'moveCount' | 'cumulativeCount'>) => string;

  moveNode: (nodeId: string, newParentId: string | null, newIndex: number) => void;

  // Getters
  getNode: (nodeId: string) => SolutionNode | undefined;

}
```

## 8. Risk Assessment

| Risk | Impact | Mitigation |
|---|---|---|
| cstimer_module browser compatibility (Web Worker) | High | Has fallback to direct import; also `scramble-display` as alternative |
| Move simplification correctness | High | FMC rules are nuanced; comprehensive unit tests; reference known solutions |
| Solution tree complexity | High | Pure function separation; well-tested tree operations; consider immutable updates |
| Touch interaction on virtual keyboard | Medium | Long-press + swipe is complex; use established gesture libraries if needed |
| State management complexity | Medium | Zustand keeps it simple; clear store separation by domain |
| Responsive layout with floating keyboard | Medium | CSS Grid/Flexbox with careful breakpoints; test on real devices |
| Large bundle size (cubing.js is 8.5MB) | Medium | Only import `cubing/alg` subpackage; tree-shake; cstimer_module is small |
